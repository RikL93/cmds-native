DEEL 2 — Native Backend Log (in-app diagnose scherm)
Bouw een nieuw scherm "Ingest Diagnose" in de native shell, bereikbaar via een knop in het bestaande diagnose-menu. Dit moet in-app werken zonder ADB.

Implementatie
Maak een ringbuffer (laatste 200 entries) in de native shell die elke ingest-actie logt naar AsyncStorage key cmds_ingest_log (JSON array). Elke entry:


{
  ts: "2026-05-02T05:36:12.123Z",
  phase: "tick" | "post_attempt" | "post_response" | "skip" | "bridge_event" | "token_refresh" | "service_state",
  detail: {
    // afhankelijk van phase, voorbeelden:
    unitId?: string | null,
    hasToken?: boolean,
    tokenExpiresIn?: number,  // seconden
    url?: string,
    httpStatus?: number,
    responseBody?: string,    // max 500 chars
    skipReason?: string,
    bridgeEvent?: "onUnitLinked" | "onUnitUnlinked" | "onSupabaseTokenRefreshed",
    bridgePayload?: object,
    serviceRunning?: boolean,
    error?: string,
  }
}
Log op deze momenten
Moment	phase	detail
Bridge event ontvangen	bridge_event	event-naam + payload
Token refresh ontvangen	token_refresh	nieuwe expiresAt
Background service start/stop	service_state	serviceRunning
Vóór elke POST-poging	post_attempt	url, hasToken, unitId, lat/lng
Na elke POST-respons	post_response	httpStatus, responseBody (eerste 500 chars)
Skip (geen unit / geen token / geen GPS)	skip	skipReason
Locatie-tick zonder POST	tick	reden waarom niet gepost
Network error	post_response	error veld gevuld
UI van het scherm
Bovenin: huidige status (3 regels):
Active unitId: <uuid of "—">
Token: <geldig nog Xs of "ontbreekt">
Service: <running|stopped>
Knoppen: Refresh · Wis log · Kopieer alle entries naar clipboard
Lijst: alle entries omgekeerd-chronologisch, met kleurcoding:
groen = post_response met status 200
rood = post_response met status ≥400 of error
geel = skip
grijs = tick / service_state / bridge_event / token_refresh
De kopieerknop levert plain-text JSON dat ik in chat kan plakken.
